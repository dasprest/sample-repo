###### 問題1  
GitHub プロジェクトでは、どのような種類のビューを作成できますか？（単一選択）

選択肢  
A. 表、ボード、ロードマップ  
B. ガント、かんばん、スプレッドシート  
C. 単純なリスト、ツリー リスト、リンクされたノード  
D. タイムライン、カレンダー、ガント、ボード  

解答  
A  

解説  
GitHub Projects（Projects v2）では、課題やプルリクエストを管理するために複数のビュー形式を作成できる。代表的なビューは **Table（表）・Board（ボード）・Roadmap（ロードマップ）** の3種類である。  

これらのビューは同じプロジェクトデータを異なる視点で表示する仕組みであり、フィルターやカスタムフィールド（Status、Priority、Iteration など）と組み合わせて柔軟に管理できる。

A. 表、ボード、ロードマップ  
正解。GitHub Projects の正式なビュータイプである。  
- **Table**：スプレッドシートのように課題を一覧管理  
- **Board**：カンバン形式でステータス管理  
- **Roadmap**：タイムラインベースで進行管理  

B. ガント、かんばん、スプレッドシート  
不正解。概念的には似ているが、GitHub Projects の正式名称ではない。  

C. 単純なリスト、ツリー リスト、リンクされたノード  
不正解。これは知識管理ツールやグラフデータ構造の概念であり、GitHub Projects のビューではない。  

D. タイムライン、カレンダー、ガント、ボード  
不正解。GitHub Projects にカレンダーやガントビューは存在しない。

要点  
GitHub Projects v2 のビューは次の3つを覚える。  

- **Table**（表ビュー）  
- **Board**（カンバンビュー）  
- **Roadmap**（タイムラインビュー）  

GitHub Foundations 試験では、この **3種類のビュー** をそのまま問われることが多い。

###### 問題2  
イテレーション フィールドはプロジェクトで何を行うのに役立ちますか?（単一選択）

選択肢  
A. issue または pull request に対して行われたさまざまな変更を追跡できる。  
B. プロジェクトに加えた変更を元に戻すことができる。  
C. プロジェクトのシーケンシャル フェーズを作成し、そのフェーズに基づいて issue と pull request をグループ化できる。  
D. 問題やプル要求に対する変更が行われると、プロジェクトを編集、繰り返し、自動化できます。  

解答  
C  

解説  
GitHub Projects の **Iteration フィールド** は、一定期間ごとの作業サイクル（スプリントなど）を管理するためのフィールドである。  

これにより、プロジェクトを **時間ベースのフェーズ（Iteration / Sprint）に分割**し、Issue や Pull Request をその期間に割り当てて整理できる。  

典型的な用途は次の通り。  

- スプリント（1週間・2週間など）の管理  
- 開発タスクの期間ベース整理  
- ロードマップ管理  
- アジャイル開発のスプリント計画  

A. issue または pull request に対して行われたさまざまな変更を追跡できる。  
不正解。変更履歴の追跡は **Issue / PR のタイムラインや Git 履歴**で管理される。  

B. プロジェクトに加えた変更を元に戻すことができる。  
不正解。これは **バージョン管理や Git のロールバック機能**の役割であり、Iteration フィールドの機能ではない。  

C. プロジェクトのシーケンシャル フェーズを作成し、そのフェーズに基づいて issue と pull request をグループ化できる。  
正解。Iteration フィールドは **時間単位のフェーズ（スプリント）を作成し、Issue / PR をその期間に割り当てるための機能**である。  

D. 問題やプル要求に対する変更が行われると、プロジェクトを編集、繰り返し、自動化できます。  
不正解。自動化は **Project workflow automation** の機能であり、Iteration フィールドとは別の概念である。  

要点  
GitHub Projects の **Iteration フィールド = スプリント管理フィールド**  

覚えるポイント  

- Iteration = 時間ブロック  
- Sprint / 開発期間の管理  
- Issue / PR を期間単位で整理

###### 問題3  
プロジェクトで優先度グループ（高・中・低など）を作成するには、どのフィールドを使用できますか？（単一選択）

選択肢  
A. 日付  
B. 単一選択  
C. イテレーション フィールド  
D. 問題点  

解答  
B  

解説  
GitHub Projects では、課題（Issue）やプルリクエスト（Pull Request）を整理するために **カスタムフィールド** を作成できる。その中でも **Single Select（単一選択）フィールド** は、あらかじめ定義した複数の選択肢から1つを選ぶ形式のフィールドであり、優先度やステータスなどの分類に適している。  

Single Select フィールドを使用すると、次のようなカテゴリを作成できる。  

- Priority: High / Medium / Low  
- Status: Todo / In progress / Done  
- Type: Bug / Feature / Enhancement  

これにより、プロジェクトのアイテムを **グループ化・フィルタリング・ソート**して整理できる。

A. 日付  
不正解。日付フィールドは期限やスケジュール管理に使用されるものであり、優先度の分類には適していない。  

B. 単一選択  
正解。Single Select フィールドは **複数のカテゴリを作成して分類する用途**に使用される。優先度グループの作成はこのフィールドの典型的な利用例である。  

C. イテレーション フィールド  
不正解。Iteration フィールドは **スプリントや時間単位の作業期間**を管理するためのフィールドである。  

D. 問題点  
不正解。Issue はタスクやバグを表す作業単位であり、分類フィールドではない。  

要点  
GitHub Projects のカスタムフィールドの中で覚えるべきもの  

- **Single Select**：カテゴリ分類（Priority / Status など）  
- **Iteration**：スプリント管理  
- **Date**：期限管理  

優先度のグループ化 → **Single Select**

###### 問題4  
プロジェクトに自動化を追加する最も簡単な方法は何ですか?（単一選択）

選択肢  
A. GraphQL API  
B. 組み込みの自動化  
C. GitHub のアクション  
D. GitHub Copilot  

解答  
B  

解説  
GitHub Projects（Projects v2）には、**Built-in automation（組み込みの自動化）**という機能があり、一般的なプロジェクト管理タスクを簡単に自動化できる。  

この機能を使用すると、次のような自動化をGUIから簡単に設定できる。  

- Issue や Pull Request が作成されたら **自動でプロジェクトに追加**  
- ステータス変更に応じて **Status フィールドを更新**  
- Pull Request がマージされたら **Done に変更**  

つまり、コードを書く必要なく **UIだけで簡単に自動化を設定できる**点が特徴である。

A. GraphQL API  
不正解。GraphQL API を使えば高度な自動化は可能だが、実装にはプログラミングが必要であり「最も簡単な方法」ではない。  

B. 組み込みの自動化  
正解。GitHub Projects に標準で用意されている **Built-in automation** が最も簡単な方法である。  

C. GitHub のアクション  
不正解。GitHub Actions は CI/CD や自動処理を実装できるが、Workflow を作成する必要があり、Projects の簡易自動化より複雑になる。  

D. GitHub Copilot  
不正解。Copilot はコード生成支援ツールであり、プロジェクトの自動化機能ではない。  

要点  
GitHub Projects の自動化は次のレベルで理解するとよい。  

1. **Built-in automation（最も簡単）**  
2. GitHub Actions  
3. GraphQL API  

試験では「最も簡単」→ **Built-in automation** を選ぶ。

###### 問題5  
プロジェクトの表示設定の変更、プロジェクトの削除、またはプロジェクトの終了に関する設定セクションの名前は何ですか?（単一選択）

選択肢  
A. 赤いゾーン  
B. 可視性とアクセス  
C. 危険ゾーン  
D. その他のオプション  

解答  
C  

解説  
GitHub の多くの設定画面には **Danger Zone（危険ゾーン）** と呼ばれるセクションがあり、プロジェクトやリポジトリに対する **重大な変更操作** を実行できる領域として用意されている。  

このセクションは通常 **赤色で強調表示**されており、誤操作を防ぐために注意を促すデザインになっている。  

Danger Zone では主に次のような操作が行える。  

- プロジェクトの **可視性変更（public / private）**  
- プロジェクトの **終了（Close / Archive）**  
- プロジェクトの **削除（Delete）**  

このように、取り消しが困難または影響が大きい操作をまとめた設定領域が **Danger Zone** である。

A. 赤いゾーン  
不正解。Danger Zone は赤く表示されるが、正式名称ではない。  

B. 可視性とアクセス  
不正解。これはアクセス制御に関する設定の概念であり、GitHub の正式な設定セクション名ではない。  

C. 危険ゾーン  
正解。GitHub の設定画面で重要な操作をまとめた正式なセクション名は **Danger Zone（危険ゾーン）** である。  

D. その他のオプション  
不正解。GitHub の設定セクション名ではない。

要点  
GitHub の設定画面で覚えるべき用語  

- **Danger Zone（危険ゾーン）**  
- 削除・公開設定変更などの **重大操作が集約された領域**

###### 問題6  
GitHub プラットフォームでは、ユーザーのパブリック アクティビティを最新の状態に保ち、通知を受け取る方法を説明します。（単一選択）

選択肢  
A. 組織をフォローする  
B. ユーザーをフォローする  
C. ユーザーのフィードをサブスクライブする  
D. 組織のフィードをサブスクライブする  

解答  
B  

解説  
GitHub では、他の開発者の活動を追跡するために **Follow（フォロー）機能**が提供されている。  

ユーザーをフォローすると、そのユーザーが行った **パブリックアクティビティ**が自分の **GitHub ダッシュボードのフィード**に表示されるようになる。  

フォローによって確認できる主なアクティビティ  

- 新しいリポジトリの作成  
- リポジトリへのスター付与  
- フォーク  
- Issue / Pull Request の作成  
- 公開リポジトリへの貢献  

これにより、他の開発者の活動を継続的に把握できる。

A. 組織をフォローする  
不正解。GitHub では主に **ユーザーをフォローする仕組み**が提供されており、組織をフォローする機能は一般的な方法ではない。  

B. ユーザーをフォローする  
正解。ユーザーをフォローすると、そのユーザーの **パブリックアクティビティの更新通知**を受け取ることができる。  

C. ユーザーのフィードをサブスクライブする  
不正解。このような機能名は GitHub の公式機能には存在しない。  

D. 組織のフィードをサブスクライブする  
不正解。GitHub の標準機能ではない。

要点  

GitHub のアクティビティ追跡の基本  

- **Follow（ユーザーをフォロー） → パブリック活動を追跡**  
- ダッシュボードの **Activity feed** に表示される  
- OSS コントリビューターや開発者の動きを追うときに利用

###### 問題7  
リポジトリをフォークする目的は何ですか?（単一選択）

選択肢  
A. アップストリーム リポジトリの複製が作成され、元のリポジトリに直接変更を加えることができます。  
B. これにより、元のリポジトリを改善する方法をユーザーが確認できる問題が自動的に作成されます。  
C. フォークを使用すると、元のリポジトリに影響を与えずにプロジェクトに変更を加えます。  
D. 変更内容をチームに表示する pull request が作成されます。  

解答  
C  

解説  
GitHub の **Fork（フォーク）** は、他人のリポジトリのコピーを自分の GitHub アカウントに作成する機能である。  

フォークを行うと、元のリポジトリ（**upstream repository**）とは独立したコピーが作成されるため、元のプロジェクトに影響を与えることなく自由に変更を加えることができる。  

一般的な OSS 開発フロー  

1. リポジトリを **Fork**  
2. フォークしたリポジトリで **変更を実装**  
3. 元リポジトリに **Pull Request を送る**  

この仕組みにより、誰でも安全に OSS プロジェクトへ貢献できる。

A. アップストリーム リポジトリの複製が作成され、元のリポジトリに直接変更を加えることができます。  
不正解。フォークはコピーを作成するが、**元リポジトリへ直接変更はできない**。  

B. これにより、元のリポジトリを改善する方法をユーザーが確認できる問題が自動的に作成されます。  
不正解。Issue は自動作成されない。  

C. フォークを使用すると、元のリポジトリに影響を与えずにプロジェクトに変更を加えます。  
正解。フォークの目的は **元リポジトリに影響を与えずに変更を行うこと**。  

D. 変更内容をチームに表示する pull request が作成されます。  
不正解。Pull Request はフォーク後に **手動で作成する**ものであり、フォーク自体の目的ではない。

要点  

Fork の本質  

- 他人のリポジトリの **コピーを自分のアカウントに作成**  
- 元のリポジトリに **影響を与えずに変更可能**  
- OSS 貢献フロー  
  **Fork → 修正 → Pull Request**

###### 問題8  
読み取りアクセス権しかない場合、pull request を作成するために必要な手順は何ですか?（単一選択）

選択肢  
A. パッチ ファイルをメンテナーに電子メールで送信します。  
B. リポジトリを複製し、変更をコミットし、プッシュを強制します。  
C. チーム メンバーとして受け入れられます。  
D. リポジトリをフォークし、複製し、変更をコミットして、フォークにプッシュします。  

解答  
D  

解説  
GitHub では、対象リポジトリに **書き込み権限（Write access）** がない場合、そのリポジトリへ直接 push することはできない。そのため、Pull Request を作成するには **Fork workflow（フォークワークフロー）** を使用する必要がある。  

一般的な手順は次の通り。  

1. 元リポジトリを **Fork**（自分のアカウントにコピーを作成）  
2. フォークしたリポジトリを **clone**  
3. 変更を **commit**  
4. 自分のフォークへ **push**  
5. 元リポジトリに **Pull Request を作成**  

この方法により、元リポジトリの権限を持っていなくても安全に変更提案を送ることができる。

A. パッチ ファイルをメンテナーに電子メールで送信します。  
不正解。GitHub では Pull Request を使用するのが標準であり、メール送信は一般的な方法ではない。  

B. リポジトリを複製し、変更をコミットし、プッシュを強制します。  
不正解。書き込み権限がない場合、push 自体ができない。  

C. チーム メンバーとして受け入れられます。  
不正解。PR 作成の前提条件ではない。  

D. リポジトリをフォークし、複製し、変更をコミットして、フォークにプッシュします。  
正解。これは **GitHub の標準的な OSS 貢献フロー** である。  

要点  

権限がないリポジトリに PR を送る基本フロー  

**Fork → Clone → Commit → Push → Pull Request**


###### 問題9  
GitHub Marketplace とは何ですか？（単一選択）

選択肢  
A. 組織の GitHub シートをさらに購入できる場所。  
B. GitHub プランをアップグレードする場所。  
C. プロジェクトのスポンサーを支援するためにスポンサーに会うことができる場所。  
D. GitHub の追加機能を検出してインストールするための場所。  

解答  
D  

解説  
**GitHub Marketplace** は、GitHub の機能を拡張するための **アプリケーション・GitHub Actions・統合ツールなどを見つけてインストールできるマーケットプレイス**である。  

Marketplace では、コミュニティや企業が開発した多くのツールが提供されており、GitHub リポジトリや開発ワークフローに追加機能を簡単に統合できる。  

代表的な例  

- **CI/CD ツール**  
- **セキュリティスキャンツール**  
- **コード品質分析ツール**  
- **プロジェクト管理ツール**  
- **GitHub Actions 拡張**

これらをインストールすることで、GitHub の標準機能に加えて **開発・DevOps ワークフローを強化**できる。

A. 組織の GitHub シートをさらに購入できる場所。  
不正解。シート（ユーザーライセンス）は **GitHub プラン管理**で行う。  

B. GitHub プランをアップグレードする場所。  
不正解。プラン変更は **Billing / Plan 設定**で行う。  

C. プロジェクトのスポンサーを支援するためにスポンサーに会うことができる場所。  
不正解。スポンサー支援は **GitHub Sponsors** の機能であり Marketplace ではない。  

D. GitHub の追加機能を検出してインストールするための場所。  
正解。Marketplace は **GitHub 拡張ツールのカタログ**である。

要点  

GitHub Marketplace  

- GitHub **拡張ツールのストア**  
- **Apps / Actions / Integrations** を提供  
- 開発ワークフローを強化するツールを追加可能

###### 問題10  
特定のユーザーまたはチームが承認しない限り、リポジトリの特定の領域に対するプル要求がマージされないようにするにはどうすればよいですか?（単一選択）

選択肢  
A. pull request ポリシーを CONTRIBUTING.md で明確に説明します。  
B. CODEOWNERS ファイルを使用し、必要なレビューを有効にします。  
C. SECURITY.md で必要なユーザーにテーブル マッピング ディレクトリ パスを追加します。  
解答  
B  

解説  
GitHub では、リポジトリ内の特定のファイルやディレクトリに対して **特定のユーザーまたはチームのレビューを必須にする仕組み**として **CODEOWNERS ファイル**を使用できる。  

CODEOWNERS ファイルでは、ファイルパスやディレクトリごとに **コード所有者（Code Owner）** を指定できる。  
さらに **Branch Protection Rule（ブランチ保護ルール）で「Require review from Code Owners」** を有効にすると、指定された Code Owner がレビューしない限り Pull Request をマージできなくなる。  

例（CODEOWNERS）  

```
/docs/ @docs-team  
/src/security/ @security-team
```

この設定により  

- `/docs` の変更 → docs-team のレビューが必須  
- `/src/security` の変更 → security-team のレビューが必須  

となる。

A. pull request ポリシーを CONTRIBUTING.md で明確に説明します。  
不正解。CONTRIBUTING.md は **開発ルールを説明するドキュメント**であり、GitHub が強制する仕組みではない。  

B. CODEOWNERS ファイルを使用し、必要なレビューを有効にします。  
正解。CODEOWNERS + ブランチ保護ルールで **レビュー必須ルールを強制できる**。  

C. SECURITY.md で必要なユーザーにテーブル マッピング ディレクトリ パスを追加します。  
不正解。SECURITY.md は **セキュリティポリシーを説明するドキュメント**であり、レビュー強制機能はない。  

要点  

GitHub のレビュー強制の仕組み  

- **CODEOWNERS** → ファイルの所有者を指定  
- **Branch protection rule** → Code Owner のレビューを必須化  

試験では  

**特定ディレクトリのレビュー必須 → CODEOWNERS**

pull request を確認するように要求されました。これに目を通すと、いくつかの軽微なコーディング エラーと入力ミスがあることに気づきます。レビューはどのように処理すればよいですか?（単一選択）

不正解。Git 操作の認証方法であり、Web ログインの仕組みではない。

D. 新しいブランチ生成  
不正解。比較はブランチ作成とは関係ない。

要点  

Pull Request のブランチ関係

###### 問題48  
このテキストを生成する Markdown スニペットを選択してください: *Hello,**world**!*（単一選択）

B. `**Hello, *world*!**`  
C. `*Hello, **world**!*`  
D. `#### Hello, world!`  

解答  
C

解説  
Markdown ではアスタリスク（`*`）を使って **斜体** や **太字** を表現できる。  

基本ルール  

- `*text*` → 斜体（italic）  
- `**text**` → 太字（bold）  

また、これらは **ネスト（組み合わせ）**することが可能である。  

今回の問題では  

- **Hello, world! 全体が太字**  
- **world が斜体**

という構造の Markdown を生成する必要がある。  

そのため  

- 外側 → `** **`（太字）  
- 内側 → `* *`（斜体）

を組み合わせる必要がある。

A. *Hello, *world*!*    
不正解。Markdown のネスト構造が壊れている。    
  
B. **Hello, *world*!**    
不正解。外側が太字、内側が斜体。    
  
C. *Hello, **world**!*    
正解。Hello が斜体、world が太字。    
  
D. #### Hello, world!    
不正解。これは **見出し（H4）** を作る Markdown。  
  
要点    
  
Markdown 強調    
  
- `*text*` → 斜体    
- `**text**` → 太字    
- **ネスト可能**

解説

```
\_example_\

```

表示結果  

\*hello\*  
\_example\_

つまり  

- `\*` → `*` を文字として表示  
- `\_` → `_` を文字として表示

C. サポートされていない  
不正解。Markdown ではエスケープ可能。  

D. そのまま書けば表示  
不正解。書式として解釈されてしまう。

要点  

Markdown 予約文字  

- `*` `_` `#` `[` `]` など  
- **`\` を使ってエスケープする**
